# 设计原则 · Design Principles
## AI项目执行规划官 v1.0.1

---

## 核心设计哲学

### 原则一：操作手册不等于计划文档

「计划文档」是描述性的（要做什么），「操作手册」是指令性的（怎么做）。
每个步骤必须是可执行的动作，有动词、有工具、有预期结果，用户读完就能直接执行。

### 原则二：断点可恢复是不可妥协的设计要求

长时间项目中途中断是必然的，不是例外。
每个 Phase 结束时的检查点和 .project_state 文件，是用户"安全网"的最小实现。

### 原则三：三不原则保护 SOP 完整性

- 不过度设计：规划的复杂度不能超过项目实际需要
- 不静默调整：不能在不告知用户的情况下改变 SOP 中的技术决策
- 不跳验证步骤：每个步骤都必须有验证方式，"执行完就行"不是合格的步骤

### 原则四：幻觉防护是模板级责任

版本号/价格/API 限制的不确定性必须通过格式强制标注，不依赖 AI 每次"记得"。
用户看到 [需用户核实] 标签，知道需要自己验证，这比提供错误信息要好得多。

## IOSE 原则应用

| 原则 | 在本 Skill 中的体现 |
|---|---|
| Independent（独立）| 可接受任意格式的 SOP/需求输入，不强依赖 Skill 02 |
| One-responsibility（单一职责）| 只生成操作手册，不写代码，不做技术决策 |
| Structured output（结构化输出）| P-T-S三层结构保证输出格式一致，便于用户执行和检索 |
| Explicit trigger（明确触发）| 触发词专属于「执行规划/操作手册」场景 |

## PAST 原则应用（激活卡片）

| 原则 | 验证 |
|---|---|
| Paste-ready（即粘即用）| 三档激活卡均可直接粘贴到任意 AI 对话 |
| Activation signal（激活信号）| 输出「✅ AI项目执行规划官已就绪」+ 材料请求菜单 |
| Self-contained（自给自足）| 激活卡包含完整的工作规则和触发词 |
| Trigger mapping（触发映射）| 触发词专属于执行规划领域，不与其他 Skill 冲突 |

## 飞轮贡献

Planner 的操作手册质量直接影响 Skill 05 的验收结果：
- P1 缺陷（步骤无验证方式）→ 建议在 Step 模板中强制要求 verify 字段
- P1 缺陷（环境配置缺失）→ 建议在工作流中将 Phase 0 明确为强制前置
- P2 缺陷（断点恢复缺失）→ 建议在每个 Phase 模板中嵌入状态保存命令
